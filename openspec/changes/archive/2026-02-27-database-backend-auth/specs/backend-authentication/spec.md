## ADDED Requirements

### Requirement: User registration
The system SHALL allow new users to create accounts with username and password.

#### Scenario: Successful registration
- **WHEN** a user submits valid registration credentials (username and password)
- **THEN** the system creates a new user account with bcrypt-hashed password
- **THEN** the system returns the created user information (excluding password)
- **THEN** the response includes HTTP status 201 Created

#### Scenario: Duplicate username
- **WHEN** a user attempts to register with an existing username
- **THEN** the system rejects the registration
- **THEN** the system returns HTTP status 409 Conflict
- **THEN** the error message indicates username already exists

#### Scenario: Invalid credentials
- **WHEN** a user submits empty username or password
- **THEN** the system rejects the registration
- **THEN** the system returns HTTP status 400 Bad Request
- **THEN** the error message specifies which fields are invalid

### Requirement: User login with JWT
The system SHALL authenticate users and issue JWT tokens for session management.

#### Scenario: Successful login
- **WHEN** a user submits correct username and password
- **THEN** the system validates credentials against bcrypt-hashed password
- **THEN** the system generates a JWT access token with 1-hour expiry
- **THEN** the system generates a JWT refresh token with 7-day expiry
- **THEN** the response includes both tokens and user information

#### Scenario: Invalid credentials
- **WHEN** a user submits incorrect username or password
- **THEN** the system rejects the login attempt
- **THEN** the system returns HTTP status 401 Unauthorized
- **THEN** the error message does not reveal which credential was incorrect

#### Scenario: Nonexistent user
- **WHEN** a user submits credentials for a username that does not exist
- **THEN** the system returns HTTP status 401 Unauthorized
- **THEN** the response is indistinguishable from incorrect password case

### Requirement: Token refresh mechanism
The system SHALL allow users to obtain new access tokens using valid refresh tokens.

#### Scenario: Successful token refresh
- **WHEN** a user submits a valid refresh token
- **THEN** the system validates the refresh token signature and expiry
- **THEN** the system issues a new access token with 1-hour expiry
- **THEN** the response includes the new access token

#### Scenario: Expired refresh token
- **WHEN** a user submits an expired refresh token
- **THEN** the system rejects the refresh request
- **THEN** the system returns HTTP status 401 Unauthorized
- **THEN** the user must log in again to obtain new tokens

#### Scenario: Invalid refresh token
- **WHEN** a user submits a malformed or invalid refresh token
- **THEN** the system rejects the refresh request
- **THEN** the system returns HTTP status 401 Unauthorized

### Requirement: Protected route authentication
The system SHALL protect API endpoints requiring authentication with JWT validation middleware.

#### Scenario: Valid access token
- **WHEN** a request includes a valid access token in Authorization header
- **THEN** the system validates the token signature and expiry
- **THEN** the system extracts user identity from token payload
- **THEN** the system allows the request to proceed to the protected endpoint

#### Scenario: Missing access token
- **WHEN** a request to a protected endpoint has no Authorization header
- **THEN** the system rejects the request
- **THEN** the system returns HTTP status 401 Unauthorized

#### Scenario: Expired access token
- **WHEN** a request includes an expired access token
- **THEN** the system rejects the request
- **THEN** the system returns HTTP status 401 Unauthorized
- **THEN** the client should attempt token refresh

#### Scenario: Invalid token signature
- **WHEN** a request includes a token with invalid signature
- **THEN** the system rejects the request
- **THEN** the system returns HTTP status 401 Unauthorized

### Requirement: Secure password storage
The system SHALL hash passwords using bcrypt with appropriate work factor.

#### Scenario: Password hashing on registration
- **WHEN** a new user registers with a password
- **THEN** the system hashes the password using bcrypt with work factor 12
- **THEN** the system stores only the hash, never the plaintext password
- **THEN** the password hash includes salt automatically generated by bcrypt

#### Scenario: Password verification on login
- **WHEN** a user attempts to log in
- **THEN** the system retrieves the stored bcrypt hash for that username
- **THEN** the system uses bcrypt compare function to validate the password
- **THEN** the comparison is timing-safe to prevent timing attacks
